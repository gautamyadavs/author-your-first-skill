Skills Primer

What makes a skill a skill?

Four ideas to carry into the build.

What it is

A packaged instruction set

A skill tells an AI assistant or agent how to handle a repeatable task, including the steps, judgment calls, templates, and examples it should follow.

Where it lives

Inside a Skills-capable platform

Skills can live in OpenAI, Claude, Gemini, or a similar platform that lets teams save reusable assistant behavior.

How it shows up

Triggered by context

The assistant uses the skill when the user's request matches the task, context, or files the skill was designed to support.

When to make one

Repeated work with standards

The best first skills are practical workflows, tool handoffs, or team standards that people already repeat and want the assistant to perform consistently.

        Your skill is ready when…
        Naming & description
        • The name is lowercase, hyphenated, and action-oriented (verb-noun).
        • The description says what the skill produces, tightly enough that the assistant can match it against a real request.
        • The description says when to use it, including the indirect phrasings real users will type.
        Activation behavior
        • The skill activates on 2 realistic prompts a real user would write.
        • The skill does not activate on 1 nearby but out-of-scope prompt.
        • The output follows the intended structure (the same sections, the same tone).
        Resource organization
        • Long examples or multi-row templates the body would otherwise have to reproduce live in references/, cited from the body.
        • Any deterministic logic the skill depends on (parsing, conversions, validation) lives in scripts/, called from the body.
        • Static files the skill returns or reuses (images, sample data, fonts) live in assets/.

        Click each one to tick it off against the skill you just built. Any miss is a real edit, not a checkbox to argue with.

        UNDER THE HOOD · LO 1

        How skills work.

        You've built a working skill. Now zoom out: here's why it works that way, and why a folder of instructions isn't just a saved prompt at scale.

        Why skills, instead of prompts or GPTs.

        Saved prompts and Custom GPTs (configurable GPT personas you switch to manually) solve part of the problem of repeatable work, but each one shifts what you do each time:

        Saved prompt

        Each time, you:

        Find it, copy, paste, edit.

        Lives in a doc or notes app. You manage finding and updating it.

        What enters contextNothing automatic. Whatever you paste in is what the agent sees, every time.

        Custom GPT

        Each time, you:

        Switch to the GPT, then ask.

        One GPT covers one task. You can't have one per workflow your team runs.

        What enters contextThe GPT's system prompt every turn, plus retrieved chunks from any knowledge files when they're relevant.

        Skill

        Each time, you:

        Just ask in normal language.

        The agent picks the right skill on its own and runs the procedure.

        What enters contextJust the name and description up front. The body loads only when the skill matches; resources load only when the body cites them. Progressive disclosure.

        Skills change what the agent does, not just what you do. Fifty skills installed add only modest discovery overhead compared to five, because the heavy lifting (the body) only loads for whichever skill the user's request actually matches.

        The skill you just built has a specific shape. The folder you installed (everything your platform's skill authoring flow packaged for you) is the basic anatomy of every skill: a required SKILL.md file and optional bundled resources.

        my-skill/ a folder on disk
        • SKILL.md Required

          Two parts: frontmatter (name + description) and body (the instructions for the agent).

        • references/ Optional

          Long examples, format specs, reference tables.

        • scripts/ Optional

          Executable code (Python, shell, and so on).

        • assets/ Optional

          Templates, fixtures, and other static files.

        Two things make this work as an architecture: which parts the agent knows about, and which parts it actually loads. Here's that split.

        When each piece loads.

        Skills load in three gated stages, a pattern called progressive disclosure. A workspace can hold dozens of skills cheaply because most stay cold until needed.

        A note on tokens: tokens are roughly the units of text the model processes; ~100 tokens is about a sentence, 5,000 tokens is about 8 pages. Smaller numbers mean faster, cheaper responses.

            KNOWLEDGE CHECK· LO 1 application MCQ TARGETS LO 1
            UNANSWERED
            A workspace has 12 skills installed. A user asks an AI assistant, "Can you help me write a polite decline to this vendor pitch?" Which of the following is true at the moment the assistant decides what to do?
            • Athe assistant loads the full SKILL.md body of every installed skill so it can compare them and pick the best fit.
              Incorrect. This is the load-everything misconception. Loading all 12 bodies would burn ~60,000 tokens every turn. At discovery, only the lightweight metadata (name + description) is loaded; the body of a skill loads only at activation, and only for the matched skill.
            • Bthe assistant has only the name and description of each skill in context, then loads the body of whichever skill's description matches the request.
              Correct. All 12 names and descriptions are loaded into context up front (~100 tokens each). the assistant matches the prompt against those descriptions in context, then loads the body of whichever skill matches, say a vendor-comms-decline skill. The other 11 stay on disk; their bodies never enter context.
            • Cthe assistant scans every file in every skill folder, including references/ and scripts/, to figure out which skill is most relevant.
              Incorrect. Resources never load before a match. Files in references/ and scripts/ only enter context during execution, after a skill activates and its body cites them. Discovery happens entirely on metadata; resources are downstream.

            EXERCISE· Failure-mode diagnostic TARGETS LO 4

            Diagnose failures.

            The three layers you just placed, description, body, resources, are also the three places a skill can fail. The symptom tells you which one. You'll diagnose someone else's skill here, then carry the same pattern back to your own when it misbehaves. Three colleagues tried Sam's weekly-update-drafter skill (a communications skill that turns rough bullet points into the team's standard weekly update). For each result, decide which layer Sam needs to edit, the same call you'll make next time your own skill produces something wrong.

            1. Resources case · 100% resources-only

              Earlier "verbatim text paraphrased" framing wasn't airtight; learner think-aloud surfaced that "just write the exact text in body prose" felt plausible. Reframed as a file-never-loaded failure: the body needs to cite a file in references/ or assets/ so the agent loads it; writing the content inline doesn't survive across runs. Body-fix feedback names this working-memory limit explicitly. Sam and Maya use the simpler "skill activated but file never loaded" surface; the 10 other role variants use structured-asset paraphrasing as a richer variation on the same KC. Placed first in the diagnostic sequence so the three correct-answer layers (resources / description / body) don't appear in their canonical introduction order; otherwise an attentive learner can pattern-match the answer key without engaging with each symptom.

            QUESTION 01 · RECALL · TARGETS LO 1
            UNANSWERED
            the assistant is about to decide whether to use your meeting-notes-summarizer skill for a user's request. Before that decision is made, which content from your skill is loaded into context?
            • AOnly the name and description from the frontmatter.
              Correct. Progressive disclosure in action: every available skill's metadata (name + description, around 100 tokens) is always loaded into context so the assistant can match the request against the catalog. Only after the match does the full body load. Everything else stays on disk until execution.
            • BThe frontmatter plus the first paragraph of the body, as a preview to help with the matching decision.
              Incorrect. There's a sharp boundary between matching and using a skill, not a graduated one. Either the assistant has activated the skill (and read the full body) or it hasn't. The first paragraph isn't a special preview tier.
            • CThe full SKILL.md including frontmatter and body, but not any files in references/.
              Incorrect. That's close to how it works after the assistant has decided to use the skill. The question asks what's loaded before that decision. Try the option that loads less.
            Q2 · length-cue inverted

            Replaced over-broad distractor C with a verbose-description trap (WHEN signals buried in 326 chars of expository padding). New C is the longest option; correct B is mid-length at 152 chars. Length-as-cue heuristic now points to a wrong answer, neutralizing IWF #4.

            IWF #4 length cue · LO 3 application
            QUESTION 02 · APPLICATION · TARGETS LO 3
            UNANSWERED
            A teammate's skill has the description: "skill for converting CSV data into clean, formatted reports." They've shipped it but it's almost never activating. Which rewrite gives the biggest gain?
            • A"Convert CSV data into clean, formatted reports for downstream analysts to review and share with stakeholders."
              Incorrect. This is WHAT only. It describes what the skill produces but says nothing about when a user would invoke it. the assistant would have to infer the situation from scratch. Look for an option that names both required pieces.
            • B"Convert a CSV into a formatted report with summary stats. Use when the user shares a CSV or asks to 'analyze', 'visualize', or 'report on' tabular data."
              Correct. Both required pieces are named: WHAT the skill does ("formatted report with summary stats") and WHEN to use it ("user shares a CSV"). The WHEN is grounded in trigger phrases users actually say ("analyze", "visualize", "report on") rather than left abstract.
            • C"Convert CSV data into a comprehensive report. This skill is for users who have CSV files and need to analyze, summarize, or visualize their tabular data. The output includes summary statistics where appropriate, formatted output for downstream review by analysts or stakeholders, and handles standard CSV formats with headers."
              Incorrect. The verbose-description trap. WHEN signals are present but buried in expository padding ("CSV files", "tabular data", "downstream review by analysts"), which dilutes the match. Every word here loads on every conversation at discovery; padding the assistant with context it already has burns tokens without improving activation. Tight WHAT plus explicit trigger phrases beats verbose context every time.
            QUESTION 03 · ANALYSIS · TARGETS LO 2
            UNANSWERED
            Your skill activates correctly on the right requests, but each time it's used the response feels noticeably slower than before you added it. Where do you look first?
            • AThe description in the frontmatter. It's the part loaded on every conversation.
              Incorrect. The description does load on every conversation, but it's tens of tokens, far too small to feel slow. The slowdown started when the skill activated, so look at the tier that loads at activation, not the one that's always there.
            • BThe body: the content that loads in full once a match is made.
              Correct. The SKILL.md body loads in full once a match is made. A body inflated with examples and edge cases pays that cost on every invocation. The fix is to move examples, edge cases, and long format specs into resources (a file in references/, scripts/, or assets/) that the body cites only when needed.
            • CThe files in references/ that the body cites mid-procedure.
              Incorrect. References do load, but only at the moment they're cited mid-procedure, not at activation. They can contribute, but the body is the more likely first culprit because it loads earlier and unconditionally once activated.
            QUESTION 04 · EVALUATION · TARGETS LO 3
            UNANSWERED
            A teammate proposes a single skill called meeting-toolkit that summarizes notes, drafts a follow-up email, and creates Linear tickets for action items. What's the deepest critique?
            • AThe name meeting-toolkit is too generic. It sounds like packaging, doesn't suggest when to invoke it, and gives no signal of what the skill produces.
              Incorrect. A real problem, but a cosmetic one. the assistant matches against the description, not the name. A vague name is a symptom of something deeper underneath. Keep digging.
            • BThe body becomes bloated once all three procedures are written out, slowing every invocation by a meaningful amount on every use.
              Incorrect. True, and you'd feel it on every use. But this is a consequence of the more fundamental issue. What's the root design flaw that makes the body bloated in the first place?
            • CThe activation contract is ambiguous: three different jobs share one description, so the assistant can't tell which one the user is asking for on any given turn.
              Correct. Three jobs share one description, so it has to be vague enough to cover all three, which means none of them activate cleanly. Three small skills each have a crisp description, and the assistant composes them across turns when the work needs all three. Composability comes for free once each skill is well-scoped.
