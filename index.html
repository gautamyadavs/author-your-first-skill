Architectural pivot: build-first, concepts-after. The lesson previously sequenced concept-first (1.1 What a Skill is → structural pre-training → placement decisions → description authoring → lab → diagnostic → KC). SME-validated drop-off data showed high learner abandonment in the opening sections: the lesson title promises "Author your first Skill" but in the prior sequence the learner did not touch authoring until well into the page. The fix flips the architecture: 1.1 (Build your first skill) puts the design workshop first then hands off to ChatGPT's skill-creator. 1.2 (Inside SKILL.md) covers the file anatomy and placement of the three layers, immediately after the lab so the file structure has a built artifact to anchor onto. 1.3 (How skills work) introduces the activation architecture as under-the-hood explanation after experience, with the simulation giving the learner a runtime view of the layers they just placed. 1.4 (Diagnose failures) applies all of that, the three layers the learner just placed are the three places a skill can fail. 1.5 is the synthesis Knowledge Check. The pedagogical bet (Keller ARCS Relevance, Bjork desirable difficulties): a learner who's built their own working thing cares about the architecture in a way they couldn't before they had personal context to anchor it onto. Mayer's pre-training principle becomes secondary when experiential context provides the anchoring scaffold. Order of LO encounter changes from 1→2→3→4 to 3→4→1→2→4 (LO 4 appears twice: in the lab's symptom-to-fix patterns and in the diagnostic's three Sam cases), but the underlying KC structure and assessment count are unchanged.

Diagnose Failures customized to learner's role + skill type. Section 1.4's three-scenario diagnostic adapts its surface content to the learner's workshop pick. When the learner selects role + process skill-type in 1.1's Design workshop, the diagnostic in 1.4 swaps Sam's universal weekly-update-drafter scenario for a role-customized skill: Marketing → Maya's campaign-brief-builder, Sales → Dion's discovery-to-next-steps, Finance → Reyna's monthly-close-narrative, Engineering → Kai's design-doc-to-plan, Operations → Sam's ops-playbook-writer, Customer Success → Jordan's qbr-storyline-builder, IT → Tomás's incident-postmortem-writer, Legal → Aviva's contract-review-summary, HR → Naia's interview-kit-builder, Management → Owen's weekly-status-multi-format, Executives → Kira's exec-decision-brief. Eleven role-process combinations covered. Tool and standards skill types fall back to Sam's universal scenario as a graceful default; the portability framing in the lede ("you'll diagnose someone else's skill here, then carry the same pattern back to your own") makes the fallback feel deliberate. Scope decision: process-type covers the most common skill kind, and the three failure-mode structures (description-failure / body-failure / resources-failure) are universal so the LO 4 KC content is identical across all customizations. Surface variation lives in: skill name, character name, three test prompts, three result narratives, and per-layer feedback HTML referencing the customized content. The resources-case framing across all 12 cases (canonical + 11 role variants): structured-asset reproduction failure. Earlier draft used "verbatim text paraphrased" (legal disclaimer for Sam, brand boilerplate for marketing, EEO statement for HR, etc.). Learner-think-aloud surfaced that the body fix was plausibly viable for several of those ("just write the exact text in body prose"), making Resources less than 100% defensible. Resolution: every resources-case item now centers on a domain-appropriate structured asset (table, matrix, or rubric) where the body alone can't reliably reproduce the structure: priority tiers for Sam and Owen, brand-palette asset for marketing, qualification rubric for sales, variance-thresholds for finance, severity rubric for engineering, SLA-tier table for operations, health-score rubric for CS, incident-severity matrix for IT, risk-flag rubric for legal, competency rubric for HR, decision rubric for executives. Each has 3 or 4 rows with cell content the model would paraphrase (definitions, thresholds, criteria, indicators); the airtight resources fix is bundle references/<asset>.md and have the body cite "copy each row exactly." The body-fix distractor's wrong-pick feedback explicitly names the working-memory failure mode: "the model can't hold a four-row rubric in working instructions reliably across runs." Architecturally: SAM object holds the default scenario; CUSTOM object keyed by 'role:type' holds the 11 customizations; resolveScenario() reads the workshop's chip state from the DOM and returns SAM or the customized entry; window.updateDiagFromWorkshop() is exposed so the workshop's selectChip can re-apply when the learner changes role or type. Items the learner has already answered correctly are preserved (locked) when the scenario re-renders; only un-answered items get the new surface content.

Feedback format standardized across all assessment surfaces. Earlier drafts used varied opening tags per assessment type: MCQs used phrasing like “That’s the load-everything misconception”, “The whole point of writing a strong description”, “Reads like a feature list”; diagnostic items used “Right”, “Not the body”, “Resources are how output gets formatted…”; workshop options used descriptive openings like “Slug-cased with a verb form”, “PascalCase isn’t the convention”; CP3 explanations had no opening tag. Even when each individual feedback string was well-written, the inconsistent openings created processing overhead: a learner scanning multiple activities had to parse a different lead-in pattern each time to figure out “am I right or wrong?” Standardized to two openings across all surfaces: <strong>Correct.</strong> for correct picks, <strong>Incorrect.</strong> for wrong picks. Both lead the feedback string so the learner’s status is the first thing they see, then the explanation follows in a uniform position. The original misconception-naming language is preserved (now appears after the standard prefix), so the substantive teaching is unchanged. Word counts audited and held to ≤60 per string. Three MCQ feedback strings (Application MCQ A and B, Concept Check B) were trimmed during the standardization pass; the rest of the strings (114 diagnostic + 12 workshop + 5 CP3 + remaining MCQs) needed only prefix changes. Pedagogically: formative-with-retry pattern preserved across all surfaces; no “the correct answer is X” reveal on wrong picks (would short-circuit the productive-struggle pedagogy these activities use). Standard applies to MCQs, workshop NAME/WHAT/WHEN decisions, diagnostic items, and CP3 drag-and-drop. Exempt: the simulation (its “feedback” is an interactive animation reveal, not a summative string) and the anatomy hotspot (no per-option text feedback, just bounce-back affordance for wrong clicks). Total ~149 feedback strings normalized.

Single-current-pick model in the workshop (revert-previous-wrong fix), extended to MCQs. After the feedback-format standardization, a learner-think-aloud audit caught an interim-state visual-noise bug specific to the workshop's apply-decision component: when a learner clicked through multiple wrong options in sequence (option A wrong, then option B wrong), both wrong-state highlights and feedback panels stayed visible simultaneously, creating a stack of red-bordered cards competing for attention. The diagnostic uses one shared feedback panel per item that gets re-populated on each click; CP3 uses bounce-back for wrong drops (no persistent feedback to revert); the sim replaces its single feedback panel on each click. The workshop and the MCQs both had per-option-independent feedback panels with no cleanup. Fix in the workshop: handleOptionClick now, before revealing the current pick's why, iterates over the decision's other options and clears the wrong class + re-hides the feedback for any previously-wrong option. A later pass extended the same model to all .question MCQs (the application MCQ and all 4 KC MCQs): the click handler now clears all other .option-feedback elements within the question before deciding what to do with the current one. Single-current-pick model: at any moment, at most one feedback panel is visible per assessment. Same-option re-click toggles (so learners can re-read); different-option click switches.

Think-aloud audit pass; 7 pain points resolved across visual, vocabulary, navigation, and auto-generated text. A first-pass learner walkthrough surfaced friction points the activity-by-activity reviews had missed because they only become visible when reading the page top-to-bottom with no prior context.

Pain point #1, LO list numbering implied a sequence that contradicts the experience order. The LO box at the top of the lesson listed the four outcomes as “01 Explain / 02 Decide / 03 Write / 04 Build,” reading visually like an ordered sequence. But the sidebar promises a build-first sequence (1.1 Design → 1.2 Build → 1.3 Diagnose → 1.4 How skills work → 1.5 Inside SKILL.md), which actually exercises the LOs in order 3 → 4 → 4 → 1 → 1+2. A learner reading the LO list at the top forms an “Explain first, Build last” expectation that the very next section, “Design YOUR skill for a real workflow”, immediately violates. Fix: dropped the ordinal markers in favor of bulleted parallel competencies. Changed <ol> to <ul> in the HTML and replaced the counter()-based numeric ::before pseudo-element with a small gold-accent dot bullet. The four outcomes are now four competencies the learner gains, not a sequence they'll experience in that order.

Pain point #4, TYPE_HINT used jargon before formal introduction. The hint that appears after the learner picks a skill-type chip used to read: “A reusable process skill encodes a recurring workflow. Most of the work lives in the body as a numbered procedure; resources stay light.” The terms body and resources are formally defined only in section 1.2 (Inside SKILL.md), but the TYPE_HINT appears in section 1.1 (before the SKILL.md anatomy is taught). A learner reading the hint to scaffold their type-pick gets jargon instead of help. Fix: rewrote all three TYPE_HINTs in plain language, naming what the skill does rather than where bits go: “The instructions are mostly a numbered procedure; the skill doesn't need many extra files” (process), “The skill talks to those systems through code rather than relying only on the model's judgment” (tool), “The skill bundles your style guides or templates so the output is consistent every time” (standards). The hints are now self-contained orientation rather than forward-references to terminology the learner hasn't met.

Pain point #5, one of six checkpoint LO labels leaked into learner view. The lesson uses <span class="checkpoint__lo lxd-only-inline">TARGETS LO N</span> to mark each activity's LO target visible only to designers (the LXD annotations toggle reveals these, learners never see them). One label on the CP3 drag-and-drop activity was missing the lxd-only-inline class, leaking “TARGETS LO 2” into the learner-facing UI. Fix: added the class. Verified all 6 .checkpoint__lo elements now have the class and are hidden from learner view.

Pain point #7, auto-generated correct WHEN option produced a period-comma typo. The workshop's correct WHEN option is constructed at runtime by appending “, or shares input that looks like [domain] content” onto starter.when. But every starter's when ends with a period (or with a period inside a closing quote, like "options on a page."), so the concatenation produced “...launch., or shares input...” or “...page.”, or shares input...”, a visible period-comma sequence that broke the option's readability. Affected all 11 roles. Fix: added a regex to strip the trailing period (preserving the trailing quote if present) before concatenating the pushy extension. Verified across all 11 roles: clean output, no typos, comma-inside-closing-quote convention preserved for the roles whose starter when ends with a quoted phrase.

Pain point #9, “pushy contexts” was lesson-specific jargon used without definition. The workshop's WHEN-decision hint, lab Step 3's iteration advice, and two auto-generated workshop feedback strings all referenced “pushy explicit contexts” or “pushy clause” as a label for the description-extension that catches user phrasings that don't directly name the task. The term is invented for this lesson and not self-explanatory. Fix: replaced across all four touch points with self-explanatory phrasing, “trigger phrases + catches for indirect requests” (workshop hint), “extend the WHEN to catch users who described their input without naming the task” (lab Step 3), “extends to catch users who describe their input rather than the task category” (workshop correct feedback), “doesn't extend to catch indirect requests” (workshop wrong feedback). The teaching is identical; the jargon is gone.

Pain point #10, diagnostic legend keeps the learner's working vocabulary, not the formal one. Earlier in the build, the diagnostic was positioned in 1.2 (immediately post-lab), before Inside SKILL.md had introduced "frontmatter," the references/ folder structure, or the formal "body" term. The legend used to read “the WHAT/WHEN in the frontmatter,” “the procedure in SKILL.md,” “templates, examples in references/” and was rewritten to plain language to avoid jargon-before-introduction. After the later architectural move (diagnostic to 1.4, after Inside SKILL.md), that precondition is gone, the learner has met all three terms by the time they reach the diagnostic. The plain-language legend was kept anyway because it reads better and reinforces the working vocabulary the workshop and lab used: Description → “the WHAT and WHEN you wrote in the workshop”; Body → “the instructions that tell ChatGPT how to do the work”; Resources → “extra files for templates or exact-wording examples the body cites.” Each entry references the learner's own workshop output or the lab's working definitions, which keeps the diagnostic anchored to what the learner did rather than what the lesson named.

Executive role article-agreement and plural-form artifacts. The workshop's auto-generated WHAT distractors used templates like 'A ' + dominantNoun + ' workflow.' (terse distractor) and 'Helps with various ' + dominantNoun + '-related workflow.' (feature-list distractor) where dominantNoun is the first token of the starter's name. Pain point #8, body-text module references contradicted the sidebar. Two Going Deeper callouts in the lesson body referenced “Module 1.2 · Multi-file Skills” and “Module 2.1 · Evaluating your Skill” using decimal cross-reference notation from an earlier version of the curriculum. The current sidebar canonicalizes top-level numbering: Module 1 = Authoring Skills (with pages 1.1-1.5), Module 2 = Multi-file Skills, Module 3 = Evaluating skills, Module 4 = Using scripts. From a learner's perspective, the body-text “Module 1.2” reads as “page 1.2 of Module 1” (which the sidebar shows as “Diagnose failures,” not Multi-file Skills), and “Module 2.1 Evaluating your Skill” refers to a sub-page that doesn't exist in the current sidebar at all. Fix applied in a later cleanup pass: the two Going Deeper callouts were removed entirely (sidebar already shows page 2 directly), and the LO-table reference to “Module 2.1 Evaluating your Skill (in the Evaluation, security & governance module)” was rephrased as “its own dedicated topic, not covered in this lesson” so the scope-discipline framing is preserved without promising a specific future deliverable the sidebar can't back up.

Pain points considered but not fixed: #2 (“starter for a skill in your role” is slightly vague upfront, resolves immediately on click, kept) and #3 (type chips have no upfront explanation, deliberate progressive disclosure since the TYPE_HINT below the chips now uses plain language; learner is not blocked).

All MCQs in the lesson audited against 19 item-writing flaws (IWF) using mcq-quality-coach; reduced 4-option items to 3. Research consensus (Rodriguez 2005 meta-analysis; Tarrant et al. 2006; NBME guidelines): three plausible distractors outperform four-with-one-weak. Each MCQ was run through scripts/validate.py for deterministic IWF checks (criteria 3, 4, 6, 9, 10, 11, 12, 14, 15, 16, 17, 19), then audited semantically against criteria 1, 2, 5, 7, 8, 13, 18. The audit originally covered six MCQs; the concept-check item documented as MCQ 2 below was subsequently removed from the lesson for content reasons (the dual-mode invocation concept it tested is now taught implicitly through the simulation's manual-vs-auto distinction and the application MCQ's discovery-stage framing, freeing one assessment slot for higher-leverage retrieval). Findings and revisions:

MCQ 1 (Application MCQ, LO 1, was 3 options): Validator flagged #4 length cue (correct answer 1.43x average distractor length). Trimmed correct option B from 269→192 chars by collapsing “the user's request to those descriptions and loads the body of whichever skill matches” to “then loads the body of whichever skill's description matches the request.” Length now within parity. Validator flags on #11 absolutes (“every installed skill” in distractors) and #12 word repeat (“user”) are intrinsic to the misconceptions being tested (load-everything, scan-the-disk) and to the scenario setup; downgraded to Pass in context.

MCQ 2 (Concept Check, LO 1, was 3 options): No revisions. Validator clean except #6 true/false-style minor risk (false positive, options are parallel completions of “How does a skill get used”). #8 logical-clue minor risk noted: A and C are direct logical complements (“only manual” vs “only auto”), B = “both.” This is the conceptual axis being tested (the misconception is “it's only one of these”); the both/only-X structure is teaching content, not a flaw.

MCQ 3 (KC Q1, LO 1, was 4 options → now 3): Validator flagged #12 word repeat (“discovery”, “content” in stem and correct answer A only). Fixed by removing “the discovery-tier content” from option A; A now reads simply “Only the name and description from the frontmatter.” Dropped option D (“The full SKILL.md and every file the body references, eagerly loaded so they're ready”) because its misconception (load-everything-eagerly) is redundant with the Application MCQ's option A which tests the same load-everything pattern at a different cognitive level.

MCQ 4 (KC Q2, LO 3, was 4 options → now 3): Validator clean on flags. The new distractor C is verbose-description (loaded WHEN signals buried in expository padding) and is longer than the correct B, which inverts the length-as-cue heuristic so test-wise learners using "pick the longest" land on a wrong option. Dropped option B (“CSV-to-report conversion. Handles various data formats, custom delimiters, multiple chart types, and several export formats”), feature-list distractor that exactly duplicates the workshop's WHAT-decision wrong-pick teaching of the packaging-copy failure. Replaced original C (over-broad “any data-related task”), the over-activation failure mode is less common in practice and its hedge words (“whenever”, “any”, “covers”) flagged as test-wise rejectable. New C tests the verbose-description misconception (more detail = better match), which is a high-frequency authoring failure mode: novice authors pad descriptions with expository context (definitions, user types, edge cases) thinking it improves matching, when in fact it dilutes the matching signal and burns discovery-time tokens on every conversation. The current distractor set: A tests no-WHEN-clause, C tests verbose-description.

MCQ 5 (KC Q3, LO 2, was 4 options → now 3): Validator flagged #12 word repeat (“skill”, “activates” appearing in stem and correct answer B only). Iteratively reworded B: “The body of SKILL.md: the content that loads in full once the skill activates” → “The body of SKILL.md: the content that loads in full once a match is made” → “The body: the content that loads in full once a match is made.” Final wording removes both cuing words; option is unambiguous given the other options reference frontmatter and references/. Dropped option D (“The total number of skills installed”) as the weakest distractor: the stem premise says the skill “activates correctly,” which directly contradicts D's misconception (more skills = slower). Test-wise learners would eliminate D on stem-grounds without engaging with the LO 2 mechanics.

MCQ 6 (KC Q4, LO 4, was 4 options → now 3): Validator flagged #12 word repeat (“single” in stem “single skill called meeting-toolkit” and correct C “share a single description”). Reworded C: “share a single description” → “share one description.” Dropped option D (“The Linear ticket creation should live in scripts/ rather than the body...”), off-topic distractor about code placement rather than multi-job activation contract. The remaining distractors (A = generic-name cosmetic critique, B = body-bloat consequence critique) both test “true but not the deepest critique” misconceptions that fit the “what's the deepest critique” stem cleanly.
